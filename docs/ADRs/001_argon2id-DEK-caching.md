**Context**: Argon2id is used for both password hashing and deriving the project_key for every read/write ops.

Every project in Vaultify has a random 32-byte salt stored in the database. When you read or write a secret, the server derives an encryption key from two things:

`project_key = Argon2id(MASTER_KEY, project.salt)`

**Problem**: Argon2id is computationally expensive. Doing it on a hot path like every read/write operation will put unnecessary load on the server.

**Solution**:
1. Initially thought of caching the derived project_key (fast, symmetric DEK) in memory/Redis, to reduce load. But that could lead to security risks, if the cache is compromised. Plus, it would lead to sync issues if I scale horizontally to multiple pods.\
**Security Flaw:** Caching increases the blast radius significantly. If a misconfigured Redis instance is compromised, the attacker gets access to all the secrets from that single DEK.

2. Decided to compute the KDF exactly once when we execute `vaultify run`, as the `project_key` is project-based not per secret based.
3. Use `golang.org/x/crypto/argon2` for both but with deliberately different parameter sets. (project_key doesn't require that much computation power as that of a low entropy human password)


**Why not use SHA256 for producing the project_key? It's fast right?**
Reasoning: SHA-256 is a general purpose hash function. It is designed to be fast and deterministic. SHA256(MASTER_KEY + salt) would technically produce a 32-byte derived key. But fast is the problem. 
Consider this: your kek_salt is stored in the database. Your MASTER_KEY is 256 bits of entropy, yes - but what if a future version of you, or a less careful operator, uses a weaker key? What if MASTER_KEY gets generated by a bad RNG? What if someone reuses a key across environments?
Argon2id as a KDF gives you defense in depth against key quality assumption.
